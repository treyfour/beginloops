## Suggested next step
Add HMAC SHA-256 signature verification to `app/api/webhooks/route.ts` — this is how you prove a webhook payload genuinely came from GitHub (or Stripe, or any provider) and not an attacker posting to your public endpoint.

```ts
// app/api/webhooks/route.ts — add this helper above the POST export

async function verifyGitHubSignature(req: NextRequest, rawBody: string): Promise<boolean> {
  const signature = req.headers.get("x-hub-signature-256");
  const secret = process.env.GITHUB_WEBHOOK_SECRET;
  if (!signature || !secret) return false;

  const key = await crypto.subtle.importKey(
    "raw",
    new TextEncoder().encode(secret),
    { name: "HMAC", hash: "SHA-256" },
    false,
    ["sign"]
  );
  const mac = await crypto.subtle.sign("HMAC", key, new TextEncoder().encode(rawBody));
  const expected =
    "sha256=" +
    Array.from(new Uint8Array(mac))
      .map((b) => b.toString(16).padStart(2, "0"))
      .join("");

  return signature === expected;
}

export async function POST(req: NextRequest) {

    return NextResponse.json({ error: "Invalid signature" }, { status: 401 });
```

Set `GITHUB_WEBHOOK_SECRET` in Vercel's environment variables dashboard, then copy the same value into GitHub → repository Settings → Webhooks → your webhook's "Secret" field. **Why this matters:** without signature verification, anyone who discovers your endpoint URL can POST arbitrary fake events. HMAC (Hash-based Message Authentication Code) is the standard pattern used by GitHub, Stripe, Twilio, and virtually every production webhook provider — the `crypto.subtle` Web Crypto API you use here is the same one available in any modern edge runtime, so learning it in Next.js App Router transfers directly.

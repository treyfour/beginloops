## Suggested next step
Add a module-level event ring buffer to `app/api/webhooks/route.ts`, then expose it via a `GET` handler — giving you a live window into the last 10 webhook events the server has received.

```ts
// app/api/webhooks/route.ts — add near the top, above the POST export

// Module-level ring buffer. Persists within a single function instance;
// resets on cold starts — that's intentional for now.
const recentEvents: { timestamp: string; event: string; data: unknown }[] = [];

export async function GET() {
  return NextResponse.json({ count: recentEvents.length, events: recentEvents });
}
```

Then inside your existing `POST` handler, push to the buffer right before you return:

```ts
// Inside POST, after parsing the event:
recentEvents.push({ timestamp: new Date().toISOString(), event: eventType, data: parsed });
if (recentEvents.length > 10) recentEvents.shift(); // drop oldest when full
```

Now hitting `GET /api/webhooks` from the browser or curl shows the live event history — no database required.

**Why this matters:** Module-level variables in serverless functions are a double-edged sword. They persist within a single warm function instance (making them fast, zero-cost storage) but reset on cold starts and aren't shared across parallel instances. Grasping this limitation is the mental bridge from "I have a variable" to "I need a database or cache (like Redis)." It's also the simplest possible event-log pattern — the same shape that drives dashboards, audit trails, and real-time activity feeds. Once you hit the cold-start wall in practice, you'll know exactly why Vercel KV or Upstash Redis exists.

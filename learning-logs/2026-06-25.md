## Suggested next step
Create a `.git/hooks/post-commit` script that pings your local heartbeat endpoint after every commit — this introduces **git hooks**, which are the local-dev equivalent of webhooks:

```bash
#!/bin/sh
# .git/hooks/post-commit
echo "→ pinging heartbeat..."
curl -s http://localhost:3000/api/cron/heartbeat | python3 -m json.tool 2>/dev/null || echo "(server not running)"
```

Make it executable: `chmod +x .git/hooks/post-commit`

Why it matters: a git hook is just a shell script that fires at a lifecycle event — `post-commit`, `pre-push`, `post-checkout`, etc. It's the same mental model as a webhook (event fires → handler runs → side effect happens), just local and shell-based instead of HTTP. Once you understand this pattern at the git level, the jump to GitHub Actions (`on: push`) and Vercel cron triggers feels obvious — they're all the same idea at different layers of the stack.

Bonus: add a `pre-commit` hook that runs `npm run lint` so your learning loop catches type errors before they hit the remote.

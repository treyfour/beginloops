Add a second Vercel cron job — `app/api/cron/digest/route.ts` — that runs weekly on Sunday mornings and logs a summary of what the service has been doing.

```ts
// app/api/cron/digest/route.ts
import { NextResponse } from "next/server";

export async function GET() {
  const payload = {
    type: "weekly-digest",
    week: getISOWeek(),
    timestamp: new Date().toISOString(),
    summary: {
      message: "Weekly digest — add real aggregation logic here as your app grows",
    },
  };

  console.log("[digest]", JSON.stringify(payload));
  return NextResponse.json(payload);
}

function getISOWeek(): number {
  const now = new Date();
  const jan4 = new Date(now.getFullYear(), 0, 4);
  const startOfWeek1 = new Date(jan4);
  startOfWeek1.setDate(jan4.getDate() - jan4.getDay() + 1);
  return Math.ceil(((now.getTime() - startOfWeek1.getTime()) / 86400000 + 1) / 7);
}
```

Then register it in `vercel.json` alongside the heartbeat:

```json
{
  "crons": [
    { "path": "/api/cron/heartbeat", "schedule": "0 16 * * *" },
    { "path": "/api/cron/digest",    "schedule": "0 9 * * 0" }
  ]
}
```

**Why this matters:** `0 9 * * 0` means "9am UTC every Sunday" — the five fields are `minute hour day-of-month month day-of-week`. Adding a second cron teaches you that Vercel runs each path independently on its own schedule, and that separating concerns (heartbeat vs. digest) into separate routes keeps each handler small and testable. Real apps typically have 3–10 cron jobs for things like sending email digests, archiving old records, syncing external APIs, and generating reports — each one a tiny stateless handler just like this.

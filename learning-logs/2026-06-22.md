Make the heartbeat cron do a real health check instead of just returning `{ status: "ok" }`. Update `app/api/cron/heartbeat/route.ts` to call your own webhook endpoint and log the result:

```ts
// app/api/cron/heartbeat/route.ts
import { NextResponse } from "next/server";

export async function GET() {
  const base = process.env.VERCEL_URL
    ? `https://${process.env.VERCEL_URL}`
    : "http://localhost:3000";

  let webhookStatus: "ok" | "unreachable" = "unreachable";
  try {
    const res = await fetch(`${base}/api/webhooks`, {
      method: "POST",
      headers: { "Content-Type": "application/json", "x-github-event": "ping" },
      body: JSON.stringify({ zen: "heartbeat probe" }),
    });
    webhookStatus = res.ok ? "ok" : "unreachable";
  } catch {
    webhookStatus = "unreachable";
  }

  const payload = {
    status: "ok",
    timestamp: new Date().toISOString(),
    checks: { webhookEndpoint: webhookStatus },
  };

  console.log("[heartbeat]", JSON.stringify(payload));
  return NextResponse.json(payload);
}
```

Why it matters: right now your cron fires daily but only proves the cron itself is alive — it doesn't verify that the rest of the app is healthy. This pattern (a scheduled job that pings other endpoints and reports their status) is the foundation of **synthetic monitoring**: you'll see it in Datadog, Checkly, and Betterstack. The `VERCEL_URL` env var is automatically set by Vercel in deployed environments, making the base URL dynamic across preview and production deployments.
